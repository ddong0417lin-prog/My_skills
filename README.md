# My Skills - Claude Code 双模型开发技能包

> 基于 Superpowers 框架增强的双模型开发技能包，包含计划审阅和代码审阅两大核心技能。

## 📦 技能列表

```
My_skills/
├── dm-plan-review/     # 双模型计划审阅
│   └── SKILL.md
├── dm-code-review/     # 双模型代码审阅
│   └── SKILL.md
└── README.md           # 本文件
```

## 🎯 技能介绍

### 1. dm-plan-review - 双模型计划审阅

**使用 Codex 对 PRD 需求分析文档和 Implementation Plan 进行双重评审。**

#### 触发场景
- 复杂功能开发前的计划评审
- 架构决策验证
- 技术方案可行性评估
- 需要多 Agent 并行协作的大型项目

#### 核心特点
| Superpowers 技能 | 集成点 |
|-----------------|--------|
| **brainstorming** | 需求澄清阶段使用苏格拉底式提问，确保需求明确 |
| **writing-plans** | 任务拆分遵循 2-5 分钟原则，拆分为可并行的小块 |

#### 执行流程

```
用户请求：/dm-plan-review 需求描述
         │
         ▼
┌─────────────────────────────┐
│  步骤一：需求澄清            │
│  - 使用 brainstorming 技能   │
│  - 苏格拉底式提问           │
└─────────────┬───────────────┘
              │ 需求明确
              ▼
┌─────────────────────────────┐
│  步骤二：编写 PRD 文档        │
│  - 创建 prd.md              │
│  - 结构化需求分析           │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  步骤三：PRD 审阅            │
│  - 调用 Codex 审阅          │
│  - 迭代优化直到通过         │
└─────────────┬───────────────┘
              │ PRD 通过
              ▼
┌─────────────────────────────┐
│  步骤四：多阶段拆分分析      │
│  - 使用 writing-plans 技能   │
│  - 拆分为 2-5 分钟小块        │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  步骤五：制定 Plan           │
│  - 创建 plan.md            │
│  - 关联 PRD 功能需求        │
└─────────────┬───────────────┘
              │
              ▼
┌─────────────────────────────┐
│  步骤六：Plan 审阅           │
│  - 调用 Codex 审阅          │
│  - 迭代优化直到通过         │
└─────────────┬───────────────┘
              │ Plan 通过
              ▼
┌─────────────────────────────┐
│  步骤七：创建 Implement.md   │
│  - 状态跟踪表格             │
│  - PRD 需求覆盖追踪         │
└─────────────┬───────────────┘
              │
              ▼
         准备就绪
      可执行 /dm-code-review
```

#### 输出文件
| 文件 | 用途 |
|------|------|
| `prd.md` | PRD 需求分析文档 |
| `prd_review.md` | PRD 审阅输入 |
| `plan.md` | 实施计划主文件 |
| `plan_review.md` | Plan 审阅输入 |
| `Implement.md` | 实施状态跟踪 |

#### 使用示例
```bash
# 完整模式
/dm-plan-review 做一个博客系统，支持 Markdown 内容管理

# 快速模式
/dm-plan-review --fast 需求描述

# 严格模式
/dm-plan-review --strict 需求描述
```

### 2. dm-code-review - 双模型代码审阅

**使用 Codex 对代码变更进行独立审查。**

#### 触发场景
- 功能实现后的代码质量检查
- 作为 dm-plan-review 的后续步骤
- 独立使用审阅代码变更

#### 核心特点
| Superpowers 技能 | 集成点 |
|-----------------|--------|
| **test-driven-development** | TDD 强制约束：RED-GREEN-REFACTOR 循环 |
| **subagent-driven-development** | 并行执行模式：多 teammate 协同开发 |
| **requesting-code-review** | 任务间审查清单：每个子任务完成后自我审查 |
| **systematic-debugging** | 四阶段调试流程：复现→定位→修复→验证 |

#### 执行模式

```
┌─────────────────────────────────────────────────────────┐
│  🔄 并行执行模式                                         │
│  → 集成 subagent-driven-development                     │
│  → 多子代理并行开发，主 Agent 协调                        │
├─────────────────────────────────────────────────────────┤
│  ✅ TDD 强制约束                                         │
│  → 集成 test-driven-development                         │
│  → RED-GREEN-REFACTOR 循环                              │
│  → 先写测试，后写实现                                    │
├─────────────────────────────────────────────────────────┤
│  🔍 任务间审查                                           │
│  → 集成 requesting-code-review                          │
│  → 子任务完成后自动审查                                  │
│  → 问题分级：严重/中等/建议                              │
├─────────────────────────────────────────────────────────┤
│  🐛 系统调试                                             │
│  → 集成 systematic-debugging                            │
│  → 四阶段根因分析：复现→定位→修复→验证                   │
└─────────────────────────────────────────────────────────┘
```

#### TDD 流程

```
┌─────────────────────────────────────────────────────────┐
│  RED: 写一个会失败的测试                                 │
│  → 先定义预期行为                                        │
│  → 运行测试，确认失败                                    │
├─────────────────────────────────────────────────────────┤
│  GREEN: 写最少的代码让测试通过                           │
│  → 不要过度设计                                          │
│  → 可以快速实现，后续再优化                              │
├─────────────────────────────────────────────────────────┤
│  REFACTOR: 重构代码，保持测试通过                        │
│  → 消除重复代码                                          │
│  → 改善代码结构                                          │
├─────────────────────────────────────────────────────────┤
│  COMMIT: 提交                                            │
│  → 等待阶段完成后统一 git commit                         │
└─────────────────────────────────────────────────────────┘
```

#### 状态流转
```
⏳ 待执行 → 🔄 进行中 → 📝 待提交 → 🔄 待审阅 → ✅ 已完成
                                    ↓
                              🔴 审阅问题 → 📝 待提交 (修复) → 🔄 待审阅
```

#### 使用示例
```bash
# 按阶段执行（需要 plan.md）
/dm-code-review

# 审阅最近一次提交
/dm-code-review --commit

# 审阅指定文件
/dm-code-review --file src/parser.ts

# 严格模式（强制 TDD）
/dm-code-review --strict
```

## 🚀 安装与使用

### 前提条件
- Claude Code 或兼容的 AI 编程助手
- Codex 访问权限（用于双重审阅）
- Superpowers 框架（推荐安装）

### 安装步骤
1. 将 `My_skills/` 目录复制到你的技能目录
2. 在 Claude Code 中加载技能
3. 开始使用 `/dm-plan-review` 和 `/dm-code-review`

### 推荐工作流
```
1. /dm-plan-review 需求描述  →  生成 PRD 和 Plan
2. 审阅并确认 Plan
3. /dm-code-review  →  开始实现
4. 按阶段执行，每阶段完成后手动触发 Codex 审阅
5. 所有阶段完成，项目交付
```

## 📝 输出文件说明

### PRD 文档 (prd.md)
- 项目概述
- 功能需求（P0/P1/P2优先级）
- 非功能性需求
- 数据需求
- 技术方案

### Plan 文档 (plan.md)
- 阶段划分
- 子任务列表（含并行性标记）
- 依赖关系
- PRD 索引映射

### Implement.md
- 实施状态跟踪表
- 子任务完成记录
- TDD 流程记录
- 任务间审查结果
- 阶段完成记录

---

## ⚠️ 注意事项

1. **PRD 是 Plan 的基础** - 必须先完成 PRD 审阅
2. **TDD 强制约束** - 所有子任务必须遵循 RED-GREEN-REFACTOR 流程
3. **阶段统一提交** - 一个阶段一个 commit，保持历史清晰
4. **用户手动审阅** - Codex 审阅需用户手动触发，主 Agent 不自动调用
5. **任务间审查** - 每个子任务完成后执行自我审查
