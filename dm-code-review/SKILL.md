# 双模型代码审阅 (Dual-Model Code Review)

## 核心原则

| 角色 | 职责 | 约束 |
|------|------|------|
| teammate | 实现代码、更新 Implement.md | **禁止 git commit** |
| 主 Agent | 协调 teammate、统一提交、串行执行、更新 Implement.md | 等待用户手动审阅反馈 |
| 用户 | 手动触发 Codex 审阅 | 反馈审阅结果 |

**关键约束**：一个阶段一个 commit，无论并行还是串行执行。

├─────────────────────────────────────────────────────────┤

│  → 集成 requesting-code-review                          │

## 执行模式选择

```
读取 plan.md → 分析阶段内子任务并行性:

┌─────────────────────────────────────────────────────────┐
│  🔄 可并行（多个 task 互不依赖）                          │
│  → Agent Team 并行执行                                   │
│  → teammate 各自实现，主 Agent 统一提交                   │
├─────────────────────────────────────────────────────────┤
│  ➡️ 串行（task 间有依赖）                                │
│  → 主 Agent 串行执行                                     │
│  → 每个 task 完成后更新 Implement.md，最后统一提交        │
└─────────────────────────────────────────────────────────┘
```

## 执行流程

### 1. 初始化阶段

```
检查 plan.md:
├─ 存在 → 按阶段执行
│   └─ 读取：阶段划分、子任务、并行性、依赖关系
└─ 不存在 → 独立模式
    ├─ --commit: 审阅最近一次提交
    ├─ --file <path>: 审阅指定文件
    └─ --batch: 批量审阅变更
```

### 2. 阶段执行循环

```
对于每个阶段:

1. 检查前置依赖是否满足
2. 执行子任务（根据并行性选择模式）
   ├─ 并行模式：启动多个 teammate 并行执行
   └─ 串行模式：主 Agent 按顺序执行
3. 统一提交（一个阶段一个 commit）
4. 等待用户手动 Codex 审阅
5. 处理反馈，通过后进入下一阶段
```

## TDD 强制约束（集成 Superpowers test-driven-development）

**所有子任务执行必须遵循 RED-GREEN-REFACTOR 循环：**

```
┌─────────────────────────────────────────────────────────┐
│  RED: 写一个会失败的测试                                 │
│  → 先定义预期行为                                        │
│  → 运行测试，确认失败                                    │
│  → 记录测试用例到 Implement.md                           │
├─────────────────────────────────────────────────────────┤
│  GREEN: 写最少的代码让测试通过                           │
│  → 不要过度设计                                          │
│  → 可以快速实现，后续再优化                              │
│  → 运行测试，确认通过                                    │
├─────────────────────────────────────────────────────────┤
│  REFACTOR: 重构代码，保持测试通过                        │
│  → 消除重复代码                                          │
│  → 改善代码结构                                          │
│  → 运行测试，确认仍然通过                                │
├─────────────────────────────────────────────────────────┤
│  COMMIT: 提交                                            │
│  → 更新 Implement.md                                     │
│  → 等待阶段完成后统一 git commit                         │
└─────────────────────────────────────────────────────────┘
```

**TDD 违反处理：**
- 如果发现 teammate 先写实现后写测试 → **删除代码，要求重写**
- 如果没有测试就直接提交 → **拒绝提交，要求补充测试**
- 测试覆盖率不达标 → **阻塞进入下一阶段**

## 并行执行模式

**适用条件**：阶段内多个 task 互不依赖，标记为 🔄 可并行

### 执行流程（集成 Superpowers subagent-driven-development）

```
主 Agent 同时启动多个 teammate:

Agent call 1: "执行子任务 1.1。
  1. 遵循 TDD 流程：先写测试，再写实现
  2. 完成后更新 Implement.md 为'📝 待提交'
  3. 禁止 git commit
  4. 记录测试覆盖的功能点"

Agent call 2: "执行子任务 1.2。
  1. 遵循 TDD 流程
  2. 完成后更新 Implement.md 为'📝 待提交'
  3. 禁止 git commit"

Agent call 3: "执行子任务 1.3。
  1. 遵循 TDD 流程
  2. 完成后更新 Implement.md 为'📝 待提交'
  3. 禁止 git commit"

↓ teammate 各自完成并更新状态

主 Agent 检测所有子任务为 📝 待提交:
→ 统一 git commit
→ 更新状态为 🔄 待审阅
→ 通知用户手动审阅
```

### teammate 执行细节（TDD + 任务间审查）

```
1. 开始：更新 Implement.md → ⏳ → 🔄 进行中

2. TDD 循环：
   RED:
   - 编写测试用例
   - 运行测试，确认失败
   - 记录测试文件路径

   GREEN:
   - 实现功能代码
   - 运行测试，确认通过

   REFACTOR:
   - 重构代码
   - 运行测试，确认仍通过

3. 任务间审查（requesting-code-review）：
   - 自我审查代码质量
   - 检查是否满足 PRD 需求
   - 记录潜在问题

4. 完成：更新 Implement.md
   - 状态：🔄 → 📝 待提交
   - 追加子任务完成记录
   - 记录测试覆盖情况
   - 注意：禁止 git commit

5. 通知主 Agent 完成
```

## 串行执行模式

**适用条件**：阶段内 task 间有依赖，标记为 ➡️ 串行

### 执行流程

```
主 Agent 按顺序执行每个 task:

执行 task A:
  1. 更新 Implement.md → task A 状态：⏳ → 🔄 进行中
  2. TDD 流程：
     - RED: 编写测试，确认失败
     - GREEN: 实现代码，测试通过
     - REFACTOR: 重构，测试仍通过
  3. 任务间审查：
     - 检查代码质量
     - 确认满足 PRD 需求
  4. 更新 Implement.md:
     - task A 状态：🔄 → 📝 待提交
     - 追加 task A 完成记录（实现内容、变更文件、问题）

执行 task B:
  1. 更新 Implement.md → task B 状态：⏳ → 🔄 进行中
  2. 实现代码、本地测试（可使用 task A 的产出）
  3. TDD 流程 + 任务间审查
  4. 更新 Implement.md:
     - task B 状态：🔄 → 📝 待提交
     - 追加 task B 完成记录

... 所有 task 完成 ...

统一提交:
  1. git add . && git commit
  2. 更新 Implement.md:
     - 所有 task 状态：📝 待提交 → 🔄 待审阅
     - 填入统一 commit hash
  3. 通知用户手动审阅
```

### 关键点

| 时机 | 操作 | 原因 |
|------|------|------|
| task 完成后 | 更新状态为 📝 待提交 | 记录进度，便于追溯 |
| task 完成后 | 追加完成记录 | 保持记录一致性 |
| 所有 task 完成后 | 统一提交 | 保持"阶段一个 commit"原则 |

**为什么串行也要"先记录待提交，后统一提交"？**
- 用户审阅时看到完整阶段变更，便于理解 task 间关系
- commit 历史清晰，每个阶段一个提交
- 与并行模式保持一致的行为模式

## 任务间审查（集成 Superpowers requesting-code-review）

**每个子任务完成后，执行自我审查：**

```
## 任务间审查清单

### 1. 功能审查
- [ ] 是否实现了 PRD 要求的功能？
- [ ] 测试用例是否覆盖了所有边界情况？
- [ ] 是否有遗漏的功能点？

### 2. 代码质量审查
- [ ] 代码是否简洁清晰？
- [ ] 是否有重复代码可以提取？
- [ ] 变量/函数命名是否规范？
- [ ] 是否有不必要的复杂度？

### 3. 技术审查
- [ ] 是否遵循技术选型？
- [ ] 接口定义是否与 Plan 一致？
- [ ] 数据结构是否正确？

### 4. 问题分级
- 🔴 严重问题：阻塞后续开发，必须立即修复
- 🟡 中等问题：建议修复，但可以记录为技术债务
- 🟢 建议优化：可选，后续迭代处理

### 5. 审查输出
记录到 Implement.md:
- 审查结果：通过 / 带问题通过 / 不通过
- 发现的问题列表
- 修复建议
```

## 系统调试（集成 Superpowers systematic-debugging）

**当遇到问题或测试失败时，使用四阶段调试流程：**

```
┌─────────────────────────────────────────────────────────┐
│  阶段 1: 复现问题                                        │
│  - 确认问题现象                                          │
│  - 编写复现步骤                                          │
│  - 确认测试能稳定复现                                    │
├─────────────────────────────────────────────────────────┤
│  阶段 2: 定位根因                                        │
│  - 使用二分法定位问题范围                                │
│  - 检查日志和错误信息                                    │
│  - 找到问题根源                                          │
├─────────────────────────────────────────────────────────┤
│  阶段 3: 修复问题                                        │
│  - 设计修复方案                                          │
│  - 实施修复                                              │
│  - 编写回归测试                                          │
├─────────────────────────────────────────────────────────┤
│  阶段 4: 验证修复                                        │
│  - 运行所有测试                                          │
│  - 确认问题已解决                                        │
│  - 确认没有引入新问题                                    │
└─────────────────────────────────────────────────────────┘
```

## 统一提交与审阅

### 提交

```bash
git add .
git commit -m "阶段 N: [阶段名称]

- 子任务 N.1: [描述]
- 子任务 N.2: [描述]
- 子任务 N.3: [描述]

TDD: 所有子任务均遵循 RED-GREEN-REFACTOR 流程
审查：任务间审查已完成
"
```

### 通知用户

```
✅ 阶段 N 已提交 (commit: abc123)

请手动让 Codex 审阅：git show HEAD
审阅完成后请将结果反馈给我。
```

### 处理审阅结果

```
用户反馈:
├─ LGTM
│   → 更新所有 task 状态：🔄 待审阅 → ✅ 已完成
│   → 追加阶段完成记录
│   → 进入下一阶段
│
└─ 有问题
    → 根据问题分配（并行模式）或直接修复（串行模式）
    → 更新状态：🔴 审阅问题
    → 修复后更新：📝 待提交 (修复)
    → 统一提交 → 🔄 待审阅
    → 再次等待用户审阅
```

## 状态流转

```
⏳ 待执行 → 🔄 进行中 → 📝 待提交 → 🔄 待审阅 → ✅ 已完成
                                    ↓
                              🔴 审阅问题 → 📝 待提交 (修复) → 🔄 待审阅
```

| 状态 | 触发条件 | 操作者 |
|------|----------|--------|
| ⏳ 待执行 | 初始状态 | - |
| 🔄 进行中 | 开始执行（TDD 开始） | teammate / 主 Agent |
| 📝 待提交 | 代码完成，等待统一提交 | teammate / 主 Agent |
| 🔄 待审阅 | 主 Agent 统一提交 | 主 Agent |
| 🔴 审阅问题 | Codex 发现问题 | 用户反馈 |
| ✅ 已完成 | Codex LGTM | 主 Agent |

| 约束 | 并行模式 | 串行模式 |

## 常见问题

**Q: 串行模式为何也要"待提交"状态？**
A: 保持"阶段一个 commit"原则，用户审阅时看到完整变更。

**Q: 串行模式 task A 完成后，task B 能看到 A 的代码吗？**
A: 能。task B 在同一工作目录执行，可直接使用 A 的产出。

**Q: 审阅失败后如何处理？**
A: 并行模式→分配给对应 teammate；串行模式→主 Agent 直接修复。统一提交后再次等待审阅。

**Q: 什么情况可以跳过 TDD？**
A: 仅限原型开发或紧急修复，需使用 `--no-tdd` 模式并记录技术债务。
