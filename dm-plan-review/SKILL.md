# 双模型计划审阅 (Dual-Model Plan Review)

## 1. 技能目标

在开始实现之前，利用 Codex 的独立审查能力对 **PRD 需求分析文档** 和 **Implementation Plan** 进行双重评审，提前发现需求理解偏差、架构问题、逻辑漏洞和边缘情况。**核心目标是将复杂任务拆分为多阶段、可并行执行的子任务，为后续 Agent Team 并行执行提供基础。**

## 2. 适用场景

- ✅ 复杂功能开发前的计划评审
- ✅ 架构决策验证
- ✅ 技术方案可行性评估
- ✅ 作为 dm-code-review 的前置步骤
- ✅ 需要多 Agent 并行协作的大型项目

## 3. 执行流程

### 步骤一：需求澄清（使用 Superpowers brainstorming）

**重要：在制定 PRD 之前，必须先确认需求是否明确。使用 Superpowers 的 brainstorming 技能进行苏格拉底式提问。**

```
用户：/dm-plan-review 需求描述

Claude:
1. 触发 brainstorming 技能进行需求澄清

2. brainstorming 核心问题清单：
   - 用户想要什么？（功能目标）
   - 为什么需要这个？（业务价值）
   - 目标用户是谁？（用户画像）
   - 核心功能有哪些？（功能列表）
   - 技术栈有要求吗？（技术约束）
   - 有参考项目/设计吗？（UI/UX 参考）
   - 边界条件是什么？（使用场景限制）
   - 非功能性需求？（性能、安全、可用性等）
   - 时间和资源约束？（项目约束）

3. 常见需要澄清的场景：
   - "做一个博客" → 问：数据源？评论系统？部署平台？用户系统？
   - "做一个管理后台" → 问：管理什么数据？用户角色？主要功能？
   - "优化性能" → 问：优化目标？当前瓶颈？预期提升？

4. 需求明确后，进入步骤二
```

**brainstorming 输出要求：**
- 需求理解摘要（200 字内）
- 核心功能列表（P0/P1/P2 优先级）
- 技术栈建议
- 关键风险和待确认事项

### 步骤三：PRD 审阅

**与 Codex 共同审阅 PRD，确保需求理解准确、完整、可行。**

```
Claude:
1. 创建 prd_review.md 文件，包含 PRD 的关键内容摘要

2. 调用 Codex 审阅 PRD：
   "作为资深产品经理和技术架构师，请评审以下 PRD 需求分析文档。
   文档详情记录在：[项目绝对路径]/prd_review.md

   请重点检查：
   1. 需求完整性
      - 是否遗漏关键功能？
      - 边界条件是否考虑充分？
      - 异常流程是否覆盖？

   2. 需求清晰度
      - 功能描述是否明确无歧义？
      - 验收标准是否可量化验证？
      - 术语使用是否一致？

   3. 需求可行性
      - 技术方案是否可行？
      - 时间和资源约束是否合理？
      - 是否存在技术风险未被缓解？

   4. 需求一致性
      - 功能需求之间是否冲突？
      - 功能需求与非功能需求是否一致？
      - 数据模型与功能需求是否匹配？

   5. 需求优先级
      - 优先级划分是否合理？
      - 是否识别了 MVP 范围？
      - 迭代计划是否合理？

   输出格式：
   ## PRD 审阅报告
   ### 严重问题 (必须解决)
   - [问题描述]

   ### 中等问题 (建议解决)
   - [问题描述]

   ### 建议优化
   - [优化建议]

   ### 审阅结论
   - ✅ 通过 / ⚠️ 带问题通过 / ❌ 不通过，需修改"

3. 根据 Codex 反馈迭代优化 PRD
4. 记录审阅统计
```

**PRD 审阅统计格式：**
```
📊 PRD 审阅统计
================
迭代次数：X
发现问题:
  - 严重：X (已解决：X)
  - 中等：X (已解决：X)
  - 建议：X (已采纳：X)

最终状态：✅ 通过 / ⚠️ 带问题通过
通过时间：YYYY-MM-DD HH:mm
```

### 步骤四：多阶段拆分分析（使用 Superpowers writing-plans）

**基于审阅通过的 PRD，进行任务拆分分析。使用 Superpowers 的 writing-plans 技能将任务拆分为 2-5 分钟可完成的小块。**

```
Claude:
1. 分析任务复杂度：
   - 简单任务（1-3 步骤）：单阶段即可
   - 中等任务（4-10 步骤）：拆分为 2-3 个阶段
   - 复杂任务（>10 步骤）：拆分为多个阶段，每阶段 3-5 个子任务

2. 阶段划分原则：
   - 按功能模块划分：不同功能模块可分阶段
   - 按依赖关系划分：有依赖的任务放在后续阶段
   - 按开发顺序划分：基础设施 → 核心功能 → 辅助功能
   - 参照 PRD 功能优先级：P0 → P1 → P2

3. 子任务拆分原则（关键）：
   - 独立性：每个子任务可独立开发、独立测试
   - 可并行：同一阶段的子任务互不依赖
   - 原子性：一个子任务对应一个明确的交付物
   - 粒度适中：每个子任务 2-5 分钟可完成（Superpowers 标准）
   - 对应 PRD：每个子任务对应明确的功能需求

4. 标注并行能力：
   - 🔄 可并行：同一阶段内可同时执行
   - ➡️ 串行：必须按顺序执行
   - 🔗 有依赖：依赖其他任务完成
```

### 步骤六：并行任务识别

**识别哪些任务可以分配给 Agent Team 并行执行，并定义进度记录规则。**

```
Claude 分析每个阶段：

1. 可并行任务组（Agent Team 模式）：
   - 识别同一阶段内无依赖的子任务
   - 这些子任务可分配给不同的 Agent teammate
   - 主 Agent 负责协调和监管

2. 串行任务组：
   - 有依赖关系的任务按顺序执行
   - 前置任务完成后才能开始后续任务

3. 进度记录规则（与 dm-code-review 对齐）：
   - 每个子任务完成后立即记录到 Implement.md
   - 不等待其他子任务完成
   - 记录内容：实现内容、变更文件、遇到的问题

4. 输出并行执行计划：

```markdown

## 9. 变体模式

### 快速模式 (/dm-plan-review --fast)
- PRD 只进行一轮审阅
- Plan 只进行一轮审阅
- 不迭代优化
- 适合快速验证想法

### 严格模式 (/dm-plan-review --strict)
- PRD 强制所有问题解决才能通过
- Plan 强制所有问题解决才能通过
- 不允许"带问题通过"
- 适合关键项目

### 仅 PRD 模式 (/dm-plan-review --prd-only)
- 只生成和审阅 PRD
- 不进行 Plan 制定
- 适合需求分析阶段

### 仅 Plan 模式 (/dm-plan-review --plan-only)
- 跳过 PRD（假设已有需求文档）
- 只生成和审阅 Plan
- 需要提供现有需求文档路径

## 10. 注意事项

- ⚠️ PRD 是 Plan 的基础，必须先完成 PRD 审阅
- ⚠️ Plan 中每个子任务必须对应明确的 PRD 需求
- ⚠️ Plan 审阅不能替代代码审阅
- ⚠️ 子任务拆分要足够独立，确保可并行执行
- ⚠️ 依赖关系标注要准确，避免并行执行时冲突
- ⚠️ 审阅文件使用绝对路径确保 Codex 可访问
- ⚠️ PRD 中的"待确认事项"在 Plan 阶段前必须解决
- ⚠️ **需求澄清阶段必须使用 brainstorming 技能**
- ⚠️ **任务拆分遵循 writing-plans 的 2-5 分钟原则**
